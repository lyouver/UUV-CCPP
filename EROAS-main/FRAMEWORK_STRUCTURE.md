# 项目框架说明（复试直白版）

## 1. 一句话讲清楚
这是一个面向水下机器人（UUV）的动态避障工程系统：
全局模块决定任务路线，局部模块负责临时绕障，控制模块负责轨迹执行。

## 2. 项目目标（先讲这个）
我的目标不是“堆最复杂算法”，而是先做出：
1. 可运行。
2. 可复现。
3. 可解释。
4. 可迭代升级。

在考研复试里，这种目标非常合理，因为它体现的是工程能力。

## 3. 系统分层
1. 仿真层：Gazebo + UUV Simulator + RexROV。
2. 感知层：动态障碍检测与速度估计。
3. 地图层：可选的占据/碰撞查询能力。
4. 规划层：全局任务路线 + 局部动态避障。
5. 控制层：6DoF 轨迹跟踪控制。

## 4. 运行流程（老师最容易理解）
1. 输入覆盖任务航点（CCPP 任务）。
2. 机器人按全局路线运行。
3. 检测到动态障碍时，局部规划器短时接管，生成避障轨迹。
4. 避障轨迹交给控制器执行。
5. 障碍解除后回到全局路线，继续任务。

核心思想：
全局管“任务连续性”，局部管“当前安全性”，两者配合而不是互相替代。

## 5. 为什么 UUV 这件事更难
1. UUV 是 6DoF，耦合更强，不是简单 2D 运动。
2. 控制是“轨迹参考驱动”，不是“给个速度就走”。
3. 中间要经过 Path/Waypoint/Trajectory 转换与坐标系对齐。
4. 这些转换环节稍有不一致，就会出现抖动、偏航、甚至失稳。

所以这个项目难点在系统协同，不只在规划器公式。

## 6. map_manager 与 OctoMap 的区别（面试回答版）
1. map_manager：更偏在线碰撞查询和局部避障联动，接入当前工程成本低。
2. OctoMap：通用地图表示能力强，适合长期建图和大场景管理。
3. 我的取舍：当前阶段优先闭环稳定和复现性，所以先用更贴合现有链路的方案。

可以这样补一句：
“不是说 OctoMap 不好，而是我先解决当前系统的主要矛盾，再做框架升级。”

## 7. 为什么不是直接 TEB / MPC / EGO（被问再答）
标准回答：
1. 我不是否定这些方法，而是按阶段推进。
2. 当前问题的主要矛盾是平台耦合稳定性，不是缺一个更复杂优化器。
3. 先把闭环做稳，再做算法替换对比，结论更可信。

如果被问“为什么不直接 costmap + MPC”：
1. 在地面机器人上这条路很常见。
2. 但在当前 UUV 平台，还要额外解决轨迹接口、TF 坐标、6DoF 控制和调参耦合。
3. 直接全上会让变量太多，阶段风险大，不利于可复现。

## 8. 你实际解决了哪些问题（正向讲）
1. 解决了局部触发不稳定：触发阈值、预测窗口、保持时间联调。
2. 解决了全局/局部交接抖动：优化切换逻辑与规划发布节奏。
3. 解决了实体通过性问题：按机器人体积做安全裕度和障碍膨胀。
4. 解决了接口链路问题：打通感知-规划-控制，形成完整闭环。

## 9. 抖动优化案例（你可重点讲）
我不是靠“一个参数玄学”，而是系统联调：
1. 降低局部轨迹重发频率，让控制器有收敛时间。
2. 无障碍时及时回归全局，避免局部长期抢控制权。
3. 平稳阶段减少额外修正，降低噪声驱动的小幅抖动。

效果：
1. 轨迹更平滑。
2. 跟踪更稳。
3. 避障和任务连续性平衡更好。

## 10. 关于速度与控制稳定（你要主动讲）
1. 水下机器人工程上强调稳定与安全，不追求激进速度。
2. 速度提升后，6DoF PID 往往需要联动重整定。
3. 我尝试过提速，但在全局-局部交接段会放大震荡。
4. 因此我选择先在合理速度范围把系统稳定性做扎实。

## 11. 这个项目体现的能力
1. 系统拆解能力：把复杂问题拆成可落地模块。
2. 工程联调能力：识别主要矛盾并做可复现优化。
3. 取舍能力：先稳定闭环，再迭代升级，而不是盲目堆算法。

## 12. 面试收尾句
这个项目的价值在于，我不仅实现了局部避障功能，还完成了水下机器人“感知-规划-控制”闭环的工程落地。
我做的是能复现、能解释、能继续升级的系统，而不是一次性的演示效果。
