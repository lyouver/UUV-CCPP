# 项目面试讲稿（直白版）

## 1. 30 秒开场（可直接背）
我的项目是一个基于 ROS1 + Gazebo + UUV Simulator 的水下机器人动态避障系统。
我做的不是单个算法，而是把感知、局部规划、全局任务、控制执行打通成一个稳定闭环。
最终实现的是：在动态障碍场景下，机器人能够稳定绕障并继续完成覆盖任务。

## 2. 90 秒扩展（老师让你讲细节）
这个项目的核心“能不能在水下平台稳定跑起来”。

我做了四件事：
1. 接入动态障碍感知，拿到动态点云和障碍速度信息。
2. 做全局与局部协同：全局给任务方向，局部负责临时绕障。
3. 做消息和坐标转换：把 Path/Waypoint/Trajectory 串起来，并处理 TF、ENU/NED 等坐标问题。
4. 做稳定性联调：把重规划频率、切换逻辑、速度边界调到可复现稳定状态。

结果是：
1. 不再是“能跑一次”，而是同场景可重复运行。
2. 局部避障和全局跟踪不再频繁抢控制权。
3. 多障碍场景下通过性更稳定。

## 3. 我到底用了什么（老师最关心）
我可以直接说：
1. 仿真平台：Gazebo + UUV Simulator（RexROV）。
2. 框架：ROS1。
3. 全局任务：覆盖任务航点（CCPP）。
4. 动态障碍输入：onboard detector 输出的动态点云与速度信息。
5. 局部避障：基于当前工程可稳定运行的反应式局部轨迹生成策略（圆弧/偏移逻辑）。
6. 控制执行：UUV 的 6DoF 轨迹跟踪控制器（DP + PID 链路）。

一句话：我不是只写了一个避障函数，而是做了完整工程闭环。

## 4. 全局-局部-控制链路（口语版，老师能听懂）
你可以这样讲：
1. 任务层先给一条全局覆盖路线（waypoints）。
2. 机器人正常情况沿全局路线走。
3. 当动态障碍进入风险范围时，局部规划器短时接管，生成一段可执行局部轨迹。
4. 这段局部轨迹不是“一个点”，而是一串带时间的参考轨迹。
5. 控制器按轨迹逐点跟踪，输出推进器需要的控制量。
6. 障碍解除后，系统退出局部接管，回到全局路线继续覆盖。

这个逻辑和地面小车常见“2D path + velocity cmd”不同。
UUV 更像“参考轨迹驱动控制”，中间有更多转换层，任何一层不一致都会抖动或失稳。

## 5. 为什么这个项目难（你要重点讲）
最大的难点不是“能不能生成轨迹”，而是“轨迹进入控制后稳不稳定”。

原因：
1. UUV 是 6DoF，姿态和位姿耦合强，不是单平面问题。
2. 局部重规划频率高时，控制器容易被频繁打断。
3. 全局和局部切换如果不干净，会出现来回抢占，表现为抖动。
4. 速度提高后，原有 PID 参数通常不再合适，需要重新整定。

所以我做的是工程取舍：先保证系统稳定闭环，再谈更重的优化器。

## 6. 你做过的关键优化（正向表达）
### 6.1 抖动治理（你可以重点讲这个案例）
我用的是“三件套联调”，不是改单一参数：
1. `plan_period` 调大：降低局部轨迹重发频率，让控制器有时间收敛。
2. 无障碍直接 `GLOBAL_PASS_THROUGH`：没有障碍就回全局，不让局部持续覆盖全局。
3. 无避障时减少额外碰撞修正：避免被地图边缘噪声驱动频繁微调。

效果是：
1. 全局跟踪段明显更稳。
2. 局部-全局切换更顺。
3. 抖动和无意义小摆动下降。

### 6.2 触发稳定性优化
1. 调整触发阈值、预测时间窗、保持时间。
2. 目标是避免“触发太晚”或“误触发太频繁”。
3. 在连续障碍场景里，第二障碍触发能力更稳。

### 6.3 安全裕度优化
1. 按机器人实体尺寸增加安全间距和障碍膨胀半径。
2. 避免“轨迹看起来能过，但实体实际过不去”。

## 7. 为什么没有直接上 TEB / MPC / EGO（被问再答）
回答原则：正面、工程化，不要防御式。

可以这样答：
1. 我不是排斥高级算法，而是按项目阶段做了正确取舍。
2. 当前任务是 CCPP 连续覆盖 + 水下 6DoF 控制链，先把闭环做稳更关键。
3. 直接上重优化器会同时引入更多变量（地图、模型、控制、调参），阶段风险高。
4. 我先建立稳定基线，后续再做 A/B 对比升级，结论更可靠。

如果老师追问“为什么不直接 costmap + MPC”：
1. 这套在地面机器人很常见，但在当前 UUV 平台还要额外处理轨迹接口、坐标转换、6DoF控制耦合。
2. 问题不只是规划器本身，而是整条执行链的稳定性。
3. 我的路线是先把系统级问题解决，再替换局部优化器，这样工程风险最低。

## 8. map_manager 与 OctoMap（直白说法）
1. map_manager：更贴近当前工程在线碰撞查询和局部避障联动，接入成本低，调试快。
2. OctoMap：通用性强，适合长期地图管理，但接入现有水下闭环还需要额外改造。
3. 我的选择逻辑：当前阶段先保证系统稳定和可复现，不追求“框架名字更大”。

## 9. 你踩过的坑（老师会觉得你做过真工程）
### 9.1 Path 到 Waypoints 转换
全局路径不能直接变成推进器命令，必须先转成可执行参考链路。
我把这条链路打通后，全局恢复才稳定。

### 9.2 TF/坐标转换
`world/world_ned`、ENU/NED、不同传感器 frame 的一致性问题非常容易出错。
这个问题前期很痛，一旦坐标链不一致，轨迹会偏、抖、甚至方向反。

### 9.3 提速导致稳定性下降
我尝试过提高速度，但速度一上去，全局-局部交接段对 PID 更敏感。
这不是“调一个参数”能解决，而是整条 6DoF 控制链都要配套整定。

## 10. 关于速度与工程实际（你可以直接背）
1. 水下平台本身速度不会像无人机那样高，工程上更重视稳定和安全。
2. 障碍速度我已经压到较低范围（比如 0.3m/s 级别），机器人速度也控制在合理范围（通常不超过 1m/s）。
3. 在这个速度区间内先把稳定性做扎实，比盲目提速更符合工程现实。

## 11. 老师高频问题（详细回答模板）
### Q1：你的核心贡献是什么？
1. 我把动态感知、局部规避、全局任务和控制执行做成了完整闭环。
2. 我解决了全局/局部切换抖动和触发不稳这些系统级问题。
3. 我形成了可复现实验和可解释调参方法，而不是“偶尔成功”。

### Q2：你这个项目的创新点在哪里？
1. 创新不只在算法公式，而在水下场景的系统整合和稳定落地。
2. 我把多模块耦合问题拆开处理并验证，这在工程上价值很高。
3. 结果是：系统从“能跑”到“可复现、可解释、可迭代”。

### Q3：你最大的困难是什么？
1. 最大困难是控制稳定性，不是轨迹生成。
2. 轨迹生成很快能做出来，但全局/局部交接和 6DoF 执行稳定要花大量联调。
3. 我最后通过节奏控制、切换逻辑和参数联调把问题压住。

### Q4：为什么你不直接用最先进方法？
1. 我不是不用，而是分阶段做。
2. 第一阶段先把稳定闭环做成，保证可复现。
3. 第二阶段再引入更复杂优化器做对照，这样结果更有说服力。

### Q5：如果再给你 3 个月你做什么？
1. 做标准化评估：碰撞率、最小距离、通过时间、轨迹平滑度。
2. 做多障碍联合决策，提升连续障碍场景通过率。
3. 在稳定基线上引入更强局部优化方法，做严谨 A/B 对照。

## 12. 一段完整叙事（3 分钟版，可直接讲）
我这个项目的目标，是在水下机器人平台上实现可运行、可复现的动态避障闭环。
一开始我也考虑过直接上更复杂的建图和优化器，但实际推进发现，当前平台下最关键的问题不是“算法名词”，而是系统稳定性。

所以我先把工程主线做扎实：
动态感知进来后，局部规划在需要时接管，不需要时回到全局任务；轨迹通过统一接口喂给控制器执行。
这个过程中最难的是全局和局部交接，以及坐标转换和控制耦合带来的抖动。

我重点做了三类工作：
第一，调重规划节奏，减少控制器被高频轨迹打断；
第二，优化状态切换，让无障碍时明确回归全局；
第三，按实体尺寸做安全裕度，避免“轨迹能过、实体过不去”。

最终结果是系统在动态障碍场景下可以稳定通过，实验可复现。
我认为这件事的价值在于：不是只证明某个算法可行，而是把一个真实机器人系统做到了可运行、可解释、可扩展。
